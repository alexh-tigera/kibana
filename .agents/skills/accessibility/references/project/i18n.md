# Localization rules (i18n)

> **Project-specific file.** This file defines the i18n contract for the current project.

## What must be localized

Every **new or changed** string that is **user-visible** or **read by assistive technology** must go through the project's i18n function:

- labels, titles, body copy
- `aria-label`, `tableCaption`, user-facing tooltip `content`
- any text exposed to screen readers

**Never** introduce these as **raw string literals**.

## What must NOT be localized

**Programmatic-only** values that are never seen or heard by users:

- radio group `name` attributes
- internal IDs, object keys
- non-assistive-technology tokens

## Default: Kibana `i18n.translate`

This project uses `i18n.translate` from `@kbn/i18n`:

```tsx
i18n.translate('messageId', { defaultMessage: 'English fallback' })
```

If the file already uses a shared i18n object (e.g. `i18nTexts.modalTitle`), follow that pattern for new strings in the same file.

### Imports

Add an `i18n` import only when you introduce a new `i18n.translate` call. Keep imports at the top and merged with existing imports from the same package.

### Message IDs

- Reuse an existing ID when the same message already exists in the file.
- Follow the local naming convention if one exists.
- Otherwise use `fileOrComponent.attribute` (e.g. `myTable.tableCaption`, `myButton.ariaLabel`).
- One ID → one `defaultMessage` per file.
- Keep `defaultMessage` short and aligned with nearby UI.

### Example

```tsx
<EuiButtonIcon
  aria-label={i18n.translate('myButton.ariaLabel', {
    defaultMessage: 'Refresh results',
  })}
/>
```
